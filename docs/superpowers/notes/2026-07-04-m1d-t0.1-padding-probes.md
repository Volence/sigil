# M1.D T0.1 — `padding` fidelity: live-asl probe matrix (F1)

**Tool:** `aeon/tools/asl` (asl 1.42 Beta Bld 212), `-cpu 68000 -q -L -U`, then `p2bin`.
**Date:** 2026-07-04. Re-runnable: sources under this doc; sensor described below.

## Why this exists

The `state.rs` header comment claimed `save`/`restore` "preserve cpu/padding/supmode
(probe-verified)". The 2026-07-04 audit (F1) found that claim **contradicted by live
asl** and unsupported by any committed probe. Per the milestone's probe-first rule, the
truth is established here against the live binary before implementation.

## The sensor

`padding on` inserts a `$00` pad byte before any word-or-larger emission (`dc.w`, `dc.l`,
a 68k instruction) that would otherwise land at an **odd** program counter. So:

```
        dc.b $AA        ; PC 0 -> 1 (odd)
        dc.w $1234      ; ON -> pad to 2: bytes AA 00 12 34 ; OFF -> AA 12 34
```

`AA 00 …` ⟹ padding **ON**; `AA 12 …` ⟹ **OFF**. Baselines confirm it:

| source | bytes | state |
|---|---|---|
| `padding on; dc.b $AA; dc.w $1234` | `aa 00 1234` | ON |
| `padding off; dc.b $AA; dc.w $1234` | `aa 1234` | OFF |
| `padding on; dc.b $AA; dc.l $12345678` | `aa 00 12345678` | ON |
| `padding on; dc.b $AA; nop` (instruction) | `aa 00 4e71` | ON — instructions pad too |
| `padding off; dc.b $AA; nop` | `aa 4e71` | OFF |

## The save/cpu/restore matrix

No bytes are emitted during setup (cpu/save/restore emit nothing), so the sensor sits at
PC 0→1 and reads the padding state at the end of the sequence.

| # | source (after `cpu 68000`) | result | padding |
|---|---|---|---|
| t12 | `padding off; save; restore` | `aa 1234` | OFF |
| t13 | `padding off; cpu z80; cpu 68000` | `aa 00 1234` | ON |
| t14 | `padding off; save; cpu z80; restore` | `aa 00 1234` | ON |
| a | `padding off; save; cpu 68000; restore` | `aa 00 1234` | ON |
| b | `padding off; save; padding on; restore` | `aa 00 1234` | ON |
| c | `padding on; save; padding off; restore` | `aa 1234` | OFF |
| d | `padding off; cpu 68000` (same cpu!) | `aa 00 1234` | ON |
| e | `padding off; cpu 68000; cpu 68000` | `aa 00 1234` | ON |
| f | `padding off; save; save; cpu z80; restore; restore` (nested) | `aa 00 1234` | ON |

## The model (hypothesis A — fits ALL rows above)

1. **`cpu X` directive resets padding/supmode to the CPU default, UNCONDITIONALLY**
   — even when `X` equals the current CPU (rows d, e). The 68000 default padding is
   **ON**. (`supmode` default is OFF; it is byte-inert — it only gates privileged-
   instruction warnings — so it does not affect ROM bytes, but is modelled for fidelity.)
2. **`save`** snapshots the CPU.
3. **`restore`** re-applies the saved CPU and, **only if that CPU differs from the
   current one**, resets padding/supmode to the restored CPU's default (rows t14, a, f).
   If the CPU is unchanged, padding/supmode are left as-is (rows t12). **`restore` never
   restores the saved padding value** — rows b and c are decisive: the saved padding
   (off / on respectively) does NOT come back; the post-`save` value survives.

The asymmetry is real: the `cpu` **directive** resets unconditionally (row d), while
`restore`'s internal CPU re-application resets only on an actual change (row t12).

## Consequence for the real Aeon ROM (the F1 point)

`main.asm:3` is `padding off`. `boot.asm`'s Z80 load blocks do
`save; cpu z80; phase …; dephase; restore`. That `restore` re-applies `cpu 68000` over a
current `z80` — an actual change — so it **resets padding to ON**. Therefore *everything
after boot.asm assembles with padding ON*, despite the `padding off` at the top. Sigil's
prior model (`restore` verbatim-restores the saved `off`) is wrong: it kept OFF.

Structs are defined early (structs.asm, before boot.asm) under padding-**off**, so their
`ds.w`/`ds.l` naive packing is unaffected by this change — the reset happens later.

The vector table (org 0, first 256 bytes) is emitted before boot.asm, under padding OFF,
so it too is unaffected — which is why the existing strict gates stay green.
